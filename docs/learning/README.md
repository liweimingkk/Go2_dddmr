# 功能学习文档规范与模板

本目录用于记录实质功能改动背后的原理、实现和验证过程，让代码交付同时成为可复习、可练习、可用于求职准备的学习材料。

实质功能改动包括新功能、非简单缺陷修复，以及算法、架构、接口、数据流或关键配置的明显变化。拼写、注释、格式、生成文件刷新、简单常量或文案调整不必单独建文档，但交付说明仍需解释改动原因和验证方式。

文件名使用 `YYYY-MM-DD-<topic>.md`。`<topic>` 使用简短、可搜索的英文 kebab-case，例如 `2026-07-20-learning-first-development.md`。

复制下面的结构创建文档。没有相关内容的可选项应写明“不适用”及原因，不要编造内容。

---

# <功能名称>

## 1. 要解决的问题

- 使用场景和用户目标是什么？
- 修改前的行为或故障是什么？
- 完成标准是什么？

## 2. 需要先懂的知识

- 列出理解该功能所需的概念。
- 用自己的话解释术语，避免只给定义。

## 3. 原理与数据流

- 描述输入、关键处理步骤、状态变化和输出。
- 必要时使用一张简短的数据流图或时序图。
- 解释“为什么这样工作”，而不只是“执行了哪些命令”。

## 4. 方案选择与取舍

- 最终采用什么方案，为什么？
- 考虑过哪些替代方案，为什么没有采用？
- 性能、复杂度、兼容性和安全性方面有什么取舍？

## 5. 代码实现导读

- 列出关键文件、入口函数、接口、ROS topic 或配置项。
- 按执行顺序说明数据和控制如何流转。
- 只引用必要的短代码片段，重点解释逻辑和设计意图。

## 6. 验证与结果

- 记录运行过的检查、测试命令及其结果。
- 覆盖正常路径、失败路径和重要边界情况。
- 将“已验证事实”“尚未验证假设”“需要现场或人工验证的内容”明确分开。

## 7. 常见问题与排查顺序

- 描述最可能出现的错误、可观察现象和定位方法。
- 给出从低成本检查到深入检查的顺序。

## 8. 动手练习

- 提供一个可以独立完成的小改动或实验。
- 写清预期现象和自检方法，但不要直接给出完整答案。

## 9. 求职准备

### 可迁移知识

- 总结可用于其他项目的工程方法或技术概念。

### 面试问题

1. 提供 3 至 5 个与本功能相关的问题。
2. 每个问题附简短答题要点，不写背诵式长答案。

### 简历项目描述草稿

- 使用“背景/问题 + 个人动作 + 技术方法 + 已验证结果”的结构写一条真实描述。
- 未经实际测量的性能提升、可靠性数字或个人贡献不得写入。

## 10. 后续学习路线

- 列出下一步值得阅读的仓库代码、官方资料或可继续完成的练习。
